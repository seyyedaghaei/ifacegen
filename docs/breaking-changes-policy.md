## Breaking changes policy

We follow semantic-versioning-like rules for public CLI behavior and generated output.
Changes may be considered “breaking” if they affect how users call `ifacegen` or what it generates.

### What counts as breaking

- **CLI flag changes**:
  - removing a flag
  - changing the meaning of an existing flag
  - changing output location/format in a way that affects downstream tooling
- **Generation changes**:
  - changing interface names
  - changing which methods are included/excluded
  - changing import/alias behavior in generated code
- **Template/content header**:
  - changing the generated-file preamble (`// Code generated by ifacegen. DO NOT EDIT.`)

### How to announce breaking changes

When you open a PR (or when you commit), include one of the following in the commit message:

- `BREAKING: ...`
- `BREAKING CHANGE: ...`

The release workflow will detect these markers and include them under **Breaking Changes** in `CHANGELOG.md`.

### When to bump versions

As a rule of thumb:

- Breaking changes => bump `MAJOR`
- New features (compatible) => bump `MINOR`
- Bug fixes (compatible) => bump `PATCH`
